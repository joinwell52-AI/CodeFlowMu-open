# PM Product Design Brief Gate

PM 在产品类任务里必须先完成产品方案设计闸门，再派发 DEV / QA / OPS 子任务。

适用场景：

- ADMIN 要求开发、合并、升级一个应用、PWA、小工具、页面、工作流或产品功能。
- 任务包含产品名称、目标用户、页面设计、交互、移动端、PWA、Gateway、上线、体验优化等内容。
- 用户只给了目标，没有给完整 PRD、UI 结构、验收路径或角色分工。

## 闸门原则

PM 是产品经理，不只是任务转发器。PM 可以把实现派给 DEV，把验收派给 QA，把发布派给 OPS，但必须先给出可执行的产品方案。

在产品方案没有成形前，PM 不应直接派发“做一个应用”“改一下页面”这类宽泛任务。

专业 UI 设计归入 PM 方案能力：当前不是新增运行时角色，而是 PM 必须借用 UI Playbook persona 完成信息架构、交互路径、视觉方向和可用性验收标准。

## 必须输出

PM 开工后，先在第一条 PM 过程记录、PM 派发给 DEV 的任务正文，或 PM 对 ADMIN 的阶段性报告中落下以下内容。内容必须足够具体，可以被 DEV 实现、QA 验收、OPS 发布。

## 派发要求

- PM 给 DEV 的任务必须引用本 brief，并明确产品名称、目录、页面结构、交互、PWA/Gateway 口径。
- PM 给 QA 的任务必须引用本 brief，并要求 QA 独立验收移动端、PWA、数据持久化和真实访问路径。
- PM 给 OPS 的任务必须引用本 brief，并明确需要验证的局域网、公网、Gateway 或静态发布路径。
- PM 最终 PM-to-ADMIN 报告必须对照本 brief 汇总：已完成、偏离点、验收证据、遗留问题、二次升级建议。

## 禁止

- 禁止只把 ADMIN 的原话复制给 DEV。
- 禁止没有产品定位就开始写代码。
- 禁止把“页面好看一点”“手机端支持一下”当成可执行需求。
- 禁止用未验证的 URL、PWA 状态或 Gateway 状态写成已完成。
- 禁止用巡检记录、接单记录替代最终项目报告。
